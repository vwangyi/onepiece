

以 ` 制品库 `为节点 git 上的代码 到 制品库 这一阶段就是 CI 持续集成。

` 制品库 ` 到 可以访问 这一阶段是 CD 持续部署。



` 制品库 `就是一个个 成品的仓库 





## CI 


jenkins 是什么

Jenkins 是负责 拉取代码 下载依赖 编译代码   通过 docker build 等命令进行 docker 打包 得到 docker 镜像。上传到制品库 这一套流水线操作。

docker 是什么

docker 镜像 不仅仅包含代码 还包含运行代码的环境比如 node 相当于 一个 服务器系统



镜像仓库是什么

镜像仓库就是 存放 docker 镜像 的仓库





方案 1:可视化操作（提供代码的 git 地址  可视化选择环境 比如 node 





## CD


k8s 是什么

k8s 集群 是 一个开源的 容器编排 平台 负责把 制品库 的 docker 镜像运行起来 

好处是 滚动发布，用户量大可以 增加 pod。

方案 1: k8s 平台 可以多台服务器运行。

方案 2：（不用 k8s ，小企业 或个人） 一台服务器运行代码。





worker节点是什么

worker 节点就是 一台电脑服务器，有内存 有硬盘。 需要付费（物理上存在的）



pod 是运行在 worker 上的 一个最小部署单元。每个 pod 上都运行了一个 docker 镜像（逻辑上存在）

（淘宝就部署在多个 pod 上 当更新的时候 滚动发布 先更新两个 pod， 而不是全部停掉更新 对于用户的感知 只是不同地区更新有延迟而已， 而不会存在有有几十秒不能访问的情况）

deployment.yaml 是 pod 的配置文件，保证始终有一定数量的 pod 在运行。



service 是什么

service 负责把收到的请求 打到 各个 pod 上，按照权重分配（也就是负载均衡



总结：

k8s复杂统筹和协调这些组件间的关系，然后确保整一个项目可以去高效的运作起来。

K8S调度到worker节点上，然后pod里面的容器由docker运行啊，然后surface就管理department就管理这些pod.

service去对外暴露这些pod的能力啊，然后确保外面可以访问到这些po，就是这几个关系啊，如果你们可以反复反复
