## 执行栈（执行上下文栈） 
- 执行栈 采用 先进后出的栈结构 来管理 执行上下文。 一开始栈底是全局执行上下文，每调用一个函数都会创建对应的函数执行上下文 压入栈底。如果递归没有结束条件，最终就会导致栈溢出。



- 执行栈是 JavaScript 引擎用来管理函数调用顺序的栈结构。它遵循 LIFO（Last In First Out，后进先出） 原则。

- 为什么需要执行栈？  js是单线程语言 一次只能做一件事，执行栈可以
1. 管理函数的执行顺序 
2. 管理多个执行上下文的创建和销毁 
3. 记录当前执行到了哪里 
4. 处理函数的返回 


执行栈和异步代码的关系 

- 执行栈里面是当前执行的函数
- 任务队列里面是等待执行的函数， 异步任务不会立即执行，而是通过事件循环进行排队。

## 执行上下文 
- 全局执行上下文 函数执行上下文 每次创建都会压入执行栈中 
- 执行上下文里面存储了 当前函数的 this 、作用域链、变量声明 函数声明、arguments、 this 作用域链 变量对象 词法环境 等等 
- 变量对象、作用域链 和 this （变量对象里面是 变量和函数声明 函数的所有形参、函数声明和变量声明 ）  函数比变量先提升
- 在执行函数前，会先创建函数对应的执行上下文，变量提升也在这个阶段。


- 每次调用函数（不是定义）时，就创建 函数执行上下文
- 执行上下文：分为 全局执行上下文 函数执行上下文 块级执行上下文 （eval()执行上下文 esm执行上下文 箭头函数没有自己的执行上下文，它使用父级的：）



- 函数执行完毕后，它的执行上下文从栈弹出，局部变量被销毁。（如果没有其他引用 垃圾回收机制会回收） 函数对象的 [[Environment]] 属性保存了定义时的词法环境
- 执行上下文就是用来记录 函数的 词法环境 this 作用域链 等等 变量对象 等等  一般有三种 全局函数 eval 执行上下文 


- 执行上下文 有两个阶段：创建阶段和执行阶段。 

执行函数前的 创建阶段。var变量初始值是 undefined  函数初始值是 一个引用地址 执行堆内存中的实际对象。 letconst的值没有初始化。 
执行函数时 的 执行阶段 ，给变量更新值 。

执行上下文 存储在 栈内存里面 

堆内存里面 是 实际的对象 函数对象 数组对象等等 

## 词法环境和变量环境
- 执行上下文里面 存储了 this 词法环境 变量环境 

- this是当前函数的this指向 
- 词法环境是 let const 函数 class 等变量  outer 指向父级词法环境 其实就是作用域链
- 变量环境是 var 变量      

词法环境和变量环境 的 最大区别是 变量初始化时机不同： 
- var变量在执行函数前 创建执行上下文就初始化为undefined

## 主线程 只执行 执行栈 栈顶的代码 
在事件循环机制下，其他线程会将达到触发条件的异步任务 放到 任务队列中排队 

主线程在事件循环机制下 循环执行栈顶的任务。栈空则等待。主线程只做一件事就是执行栈顶的代码 不会做其他事。

在整个执行过程中 栈底一定是 全局上下文，栈顶就是不同的函数切换执行。执行完毕就出栈，

事件循环负责 把 任务队列中 正在排队的任务 放进 执行栈中 创建执行上下文 。

队列是有优先级的 队列里面的任务就是排队的顺序。

主线程 负责 执行栈 的 出栈 压栈 ， 执行栈的规则是 先进后出

## 编译阶段 和 执行阶段

- 编译阶段
    1. 词法分析（Lexing）：代码 → Token 流
    2. 语法分析（Parsing）：Token → AST（抽象语法树）
    3. 作用域确定：确定每个标识符在哪个作用域
    4. 生成字节码或机器码
    5. 【关键】创建全局执行上下文（结构确定，部分初始化）
- 执行阶段
    1. 逐条执行代码
    2. 遇到 var x = 5 → 赋值
    3. 遇到函数调用 → 创建新的执行上下文
    4. 逐行执行直到结束

## 变量提升和暂时性死区
- var变量 可以在声明前访问 是 在执行上下文里面 已经存储了 初始值为undefined的var变量 
- let/const 变量 存在暂时性死区是 ？ 


- var变量可以在声明前访问，是因为执行上下文里面记录了 var变量 并且初始化值为 undefined 并可以访问 没有死区。
- letconst变量 不能在声明前访问，，在执行上下文里面 也记录了let变量 但没有初始化为undefined，并且在赋值前都属于死区。访问就报错。

这样设计的好处是 避免 var 带来的提升混乱，强制变量 遵循 先声明 后使用的规范。


## 堆和栈 

- 当打开浏览器时，会分配浏览器进程，浏览器进程里面 有一块内存空间和至少一个线程。
- 内存空间就分为 堆内存和栈内存
- 栈内存 存储 基本数据类型和 指向堆内存的引用数据类型的地址（只是地址）
    - 函数调用时创建执行上下文（Execution Context）
    - LIFO（后进先出）结构
    - 函数执行完通常会被销毁 不需要垃圾回收机制介入 
- 堆内存 存储    
    - 存储引用类型的实际数据（对象、数组、函数体等）
    - 需要手动或自动（垃圾回收）管理 需要垃圾回收机制管理

    注意：函数内部的基本数据类型，虽然是基本数据类型 但因为它是函数内的局部变量且有嵌套函数引用了它，所以不在栈上，而是分配在堆中。

## 词法环境 

- 每个函数执行时，会创建自己的执行上下文，包含三部分：

```js
ExecutionContext = {
  ThisBinding: this值,
  LexicalEnvironment: {  // 词法环境
    EnvironmentRecord: {},  // 存储变量和函数声明
    outer: reference  // 指向外部词法环境（作用域链核心）
  },
  VariableEnvironment: {}
}
```
- 
```js
function outer() {
  let count = 0;        // ① 在堆中分配
  
  function inner() {    // ② inner函数体也在堆中
    count++;            // ③ 引用外部变量
    console.log(count);
  }
  
  return inner;         // ④ 返回inner函数的引用
}

const fn = outer();     // ⑤ 外部变量fn持有inner引用
fn();                   // ⑥ 执行inner
```
关键点：count 虽然是基本类型，但因为它是函数内的局部变量且有嵌套函数引用了它，所以不在栈上，而是分配在堆中。
