# 前端开发规范文档 


0. 核心目录
  - api
  - store是统一管理状态
  - hooks是复用有状态逻辑
  - utils是复用无状态逻辑
  - components是展示组件，负责UI展示和交互 不能出现业务逻辑。
  - views 容器组件 负责调用展示组件和调用store

  - views下components是私有组件 可以包含业务，如果这些业务逻辑需要复用或拆分，那就提取到hooks里面。组件需要共享提取到components里。
  - views不允许引用其他页面的私有组件 需要共享 提取到components下
  - src/components src/views  扁平化管理只有2层 不能嵌套过深
  - 大量情况私有组件很少 都要共享 


1. 核心数据必须用store统一管理，保证单一数据源。
  - 不允许用provide/inject。provide/inject用于封装跨代的父子组件 （如 `<Form>` 向 `<FormItem>` 传上下文）时使用，禁止在业务开发中用于共享状态。
  - 不允许用 reactive() 或 hooks 模拟store。统一采用pinia。
  - 必须明确的是 
  - 为什么有store这一层，因为页面一定是组件化，多个组件组成。特别是大型的B端项目一个页面的业务逻辑超级复杂。一定是模块化分组件的，用store解决组件间通信问题
  - 《一个更简单直接的解决方案是抽取出组件间的共享状态，放在一个全局单例中来管理。这样我们的组件树就变成了一个大的“视图”，而任何位置上的组件都可以访问其中的状态或触发动作。》 出自Vue官网状态管理：https://cn.vuejs.org/guide/scaling-up/state-management.html 

  - store维护的是 整个应用最核心的视图对象（方便团队所有人一起维护）
  - 什么数据进入store：页面上渲染的通过后端接口获取的数据都是重要的业务数据 都应该进入store管理
  - 什么数据由组件内部管理：有些局部UI状态（如弹窗显隐、loading状态）通常更适合放在组件内部(ref),避免store变得臃肿。
  - 虽然某些数据不应该进入store 但有时为了共享也可以进入store（取舍问题 没有银弹） 


2. 必须遵循 单向数据流   
  - Model层： store 状态：管理数据和业务逻辑
  - View层： template 视图：管理界面
  - ViewModel层： script setup 交互：连接model和view的桥梁
  - 全局单向数据流：状态变化 视图自动更新，视图通过ViewModel通知store修改数据 （不允许直接修改state）
  - 父子单向数据流：数据从父流向子，子组件通过事件通知父组件修改数据 （不允许直接修改props）
  - 跨代单向数据流：provide/inject  
  - 当传递一个引用数据类型时，只改值不改地址也算修改 如 props.list.push(item)
  
3. 容器组件和展示组件相分离。

4. 拆分模块 遵循高内聚低耦合 
  - 拆分文件 拆分函数 等等
  - 一个函数只做一件事 遇到其他事情抽离出去
  - vue文件大于200行建议拆分（检查是否职责过多） 超过500行 必须拆分 （行数只是一个警示信号 真正决定是否拆分的是高内聚低耦合）
  - 高内聚就是 模块内部成员之间内聚程度高 所有成员都为了一个目标努力，缺一不可，一个模块只做一件事。
  - 模块内部 所有成员只做1件事（内聚程度高），所有成员做了10件不相干的事（内聚程度低） 

  - 低耦合是 模块外部，模块和模块之间耦合程度低。   
  - 修改一个模块或删除这个模块 影响0个模块（耦合度低）， 影响10个模块（耦合度高）

 
5. 有的人可能有疑惑, 不共享的数据为什么存store ？
    - 理论上前端所有东西都可以写在views页面上， 比如 组件 hooks 工具函数 api 核心业务数据，这样导致页面臃肿难以维护
    - 所以分门别类的 《把 组件、hooks、工具函数、api、核心业务数据 从页面上抽离出来》 
    - 那么抽离到store主要是为了 更好地维护 更加高内聚低耦合 让页面干净清爽
    - 如果你认为 这个数据 永远不会共享 放页面上也不会臃肿 那么放页面上也没关系 （取舍问题）







- vue 统一拥抱 `<script setup>`, 放弃 optionAPI 或 setup()函数
- store写法 拥抱Setup Store， 放弃 Option Store写法  
- 命名规范 components 和 views/components 命名使用大驼峰 且2-3个单词，拥抱 XxxXxx/XxxXxx.vue 放弃 XxxXxx/index.vue 
- 优先async/await 
- 拒绝魔法数字
- 变量声明：无脑const 修改用let 不用var
- 保持低圈复杂度 圈复杂度值必须小于10, 大于10必须重构。代码中“分支判断”越多，圈复杂度越高。 3 选 switch，5 上 map，再多上策略
- 避免长函数（函数要符合单一职责原则）

## 高内聚
高内聚就是 模块内部成员之间内聚程度高 所有成员都为了一个目标努力，缺一不可，一个模块只做一件事。
模块内部 所有成员只做1件事（内聚程度高），所有成员做了10件不相干的事（内聚程度低）

## 低耦合


低耦合是 模块外部，模块和模块之间耦合程度低。   
修改一个模块或删除这个模块 影响0个模块（耦合度低）， 影响10个模块（耦合度高）


模块之间不通信是最好的， 只传1个参数 比 要传2个参数要好， 模块之间尽量不要通信。




## store
```js
import { ref, computed } from "vue";
import { defineStore } from "pinia";
import  {getModeList} from "epg-api";




// 处理数据：把 数据传输对象 转为 视图对象
function transform(data) {
  const list = data?.data?.result?.[0]?.cells || [];
  return list.map((item) => {
    return {
      ...item, // 保留原有数据 不要修改原有数据 
      isSelected: false, // 是否选中 
      title: item?.props?.title || "", // 标题
      img: item?.props?.img || "", // 背景图片
      description: item?.props?.description || "", // 描述
      // ... 扩展其他业务数据
    };
  });
}



/* 命名规范小驼峰：`use${模块名}Store` */
export  const useDataStore = defineStore("data", () => {
  // 封装：把 数据 和 操作数据的方法 封装在一起。  《重要 重要 重要》

  /* ========== 定义 view object 的数据结构 给页面上访问 ============= */
  const modeList = ref([]); // 列表数据  isSelected 选中状态包含在原数据里面  也可以新增一个变量记录选中项 各有优劣 自行取舍
  const currentMode = computed(() => modeList.value.find((v) => v.isSelected) || {}); // 选中项
  
  /* ========== 定义操作 view object 的方法 给页面上触发 ============= */
  async function initModeList() {
    const res = await getModeList();
    modeList.value = transform(res); // 核心数据进入store 进行管理
    return res?.msg || '操作成功'; // 后端给的一些不重要的数据不要进入store 比如提示信息 可以返给页面，进行提示
  }

  /**
   * @description 设置选中状态
   * 目前业务逻辑很简单，点击谁就选中谁，而且是单选
   */
  function setSelected(item) {
    modeList.value = modeList.value.map((v) => ({
      ...v,
      isSelected: v.id === item.id,
    }));
  } 

  return { modeList, currentMode, initModeList, setSelected };
});
```

## page
```js
<script setup>
import { ref, computed, onMounted } from "vue";
import { useDataStore } from "@/stores/data";
import MovableFocus from "@/components/MovableFocus/MovableFocus.vue";
import VerifyDialog from "./components/VerifyDialog/VerifyDialog.vue";



const dataStore = useDataStore();
const list = computed(() => dataStore.modeList);
const showPermissionDialog = ref(null);


function handleItemFocused(item) { }

async function handleItemClicked(item) {
    // 不需要校验走then 需要校验才走catch
    await isCheckNeeded(item).catch(() => showPermissionDialog.value = item);
    dataStore.setSelected(item);
}

/**
 * 是否进行校验权限
 */
function isCheckNeeded(item) {
    return new Promise((resolve, reject) => {
        if (dataStore.currentMode.type === "chilren" && item.type !== "chilren") {
            // 离开儿童模式
            reject();
        } else {
            resolve();
        }
    });
}

function passed(flag) {
    if (flag) {
        // 通过了校验 选中当前项
        dataStore.setSelected(showPermissionDialog.value);
    }
    // 不管通过不通过 都把弹窗关掉
    showPermissionDialog.value = null;
}

onMounted(() => {
    // 初始化数据
    dataStore.initModeList();
});
</script>

<template>
    <div class="home-page">
        <!-- 
            校验弹窗 
            组件通过passed事件告诉我 是否通过校验 
        -->
        <VerifyDialog v-model="showPermissionDialog" @passed="passed"></VerifyDialog>
        <!-- 数据交给展示组件进行渲染 由展示组件告诉容器组件 用户点击了谁 聚焦了谁-->
        <MovableFocus :data="list" @item-click="handleItemClicked" @item-focus="handleItemFocused"></MovableFocus>
    </div>
</template>
```
