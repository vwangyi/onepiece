## 回答

---

### 推荐回答（标准版）

> `package.json` 中的 `version` 字段用于声明当前**包的版本号**，遵循 **语义化版本控制规范（Semantic Versioning，简称 SemVer）**，格式为 `major.minor.patch`：
> - **major（主版本号）**：做了不兼容的 API 修改时递增（破坏性变更）
> - **minor（次版本号）**：新增功能，且向下兼容时递增
> - **patch（修订号）**：向下兼容的问题修复时递增
>
> 除此之外，还可以附加预发布标签，如 `1.0.0-alpha.0`、`2.0.0-beta.1`、`3.0.0-rc.2`。
>
> 在开发和发布 npm 包时，npm 会依据这个版本号来管理包的更新与依赖安装。在 `dependencies` 中，我们也会用 `^1.2.3`、`~1.2.3` 等方式指定允许升级的版本范围，这些都是基于版本号规范的。

---

### 进阶加分点（高阶面试）

> 除了基础的三位版本号，我还想补充几个实际工程中容易忽略的关键点：
> 1. **版本锁与依赖地狱**：SemVer 只是一个**契约**，实际中很多包并未严格遵守，导致 `^1.2.3` 升级后项目崩溃。因此 CI 中常常使用 `package-lock.json` 锁定版本，或用 `npm ci` 代替 `npm install`。
> 2. **发布前版本管理**：可用 `npm version patch/minor/major` 自动升级版本并打 git tag，结合 CI/CD 完成自动发布。
> 3. **常见版本范围写法**：
>    - `^1.2.3`：兼容 `1.x.x` 的最新版（主版本不变）
>    - `~1.2.3`：兼容 `1.2.x` 的最新版（次版本不变）
>    - `>=1.2.3 <2.0.0`：显式范围
>    - `1.2.x`：通配符
> 4. **零主版本（0.y.z）的特殊性**：根据 SemVer 规则，`0.y.z` 版本是**初期开发**，任何变化都可能不兼容；实际项目中依赖 `0.x.x` 包需格外小心，最好精确锁定版本。
> 5. **工程实践**：我们团队要求每次发版必须严格遵循 SemVer 规范，并通过 CI 检查是否有 breaking change 的描述；同时在 CHANGELOG 中记录影响范围。

---

### 速记口诀（便于自己表达）

- **格式**：大.中.小
- **大**：不兼容改
- **中**：加功能兼容
- **小**：修 bug 兼容
- **前标**：alpha/beta/rc
- **范围**：`^` 大不动，`~` 中不动

### 示例（可视化）

```json
{
  "name": "my-lib",
  "version": "2.3.1"
}
```

| 版本变化 | 类型 | 例子 |
| --- | --- | --- |
| 破坏性 API 修改 | major | 2.3.1 → 3.0.0 |
| 新增功能（兼容） | minor | 2.3.1 → 2.4.0 |
| Bug 修复（兼容） | patch | 2.3.1 → 2.3.2 |
| 内测预览版 | pre-release | 2.3.1 → 2.4.0-alpha.0 |
