## Spring的三层架构
 
简介： Spring MVC 三层架构（表现层、业务层、数据访问层）通过职责分离提升代码可维护性与扩展性。
表现层（Controller）接收请求并返回响应；
业务层（Service）处理核心逻辑与事务；
数据访问层（Mapper）负责数据库操作与数据映射，共同实现高效、清晰的系统开发。
Spring MVC 的三层架构（表现层、业务层、数据访问层）是一种经典的软件设计模式，其核心作用是分离关注点，提高代码的可维护性、可扩展性和可测试性。

以下是各层的具体作用和职责
三层架构分别是

1. 表现层(Controller)：接收请求，调用 Service 处理业务逻辑。
2. 业务层(Service)：执行业务逻辑，调用 Mapper 访问数据库。
3. 数据访问层(Mapper)：执行 SQL 查询，返回结果给 Service。

## 表现层(Controller):
接收请求：处理客户端（如浏览器、APP）的 HTTP 请求。
参数解析：将请求参数（如 URL 参数、表单数据）转换为 Java 对象。
调用业务逻辑：将请求转发给对应的 Service 处理。
返回响应：将处理结果封装为视图（如 HTML）或数据（如 JSON）返回给客户端。
## 业务层(Service):
实现业务逻辑：处理核心业务需求（如订单支付、用户认证）。
事务管理：保证数据一致性（如跨表操作的原子性）。
业务规则校验：验证数据合法性（如年龄限制、金额范围）。
服务编排：协调多个 DAO 完成复杂业务流程。
## 数据访问层(Mapper):
数据持久化：将业务对象保存到数据库。
数据库操作：封装 SQL 查询（如查询用户列表、统计订单数）。
结果映射：将数据库记录转换为 Java 对象。



## ----
在 Koa2 项目中，并没有一个强制规定的目录结构，但为了项目的可维护性和可扩展性，通常会采用一种分层的架构模式。最常见的是 **MVC (Model-View-Controller)** 架构的变体，它将业务逻辑、数据模型和请求处理分离开来，让代码结构更清晰。

一个典型的、适用于中大型项目的 Koa2 目录结构如下：

```
project-root/
├── node_modules/         # 存放项目依赖的 npm 模块
├── public/               # 存放静态资源，如图片、CSS、JavaScript 文件
├── src/                  # 项目核心源代码目录
│   ├── config/           # 配置文件，如数据库连接、环境变量等
│   ├── controllers/      # 控制器层，负责处理请求和响应
│   ├── middleware/       # 自定义中间件，如身份验证、日志记录
│   ├── models/           # 模型层，定义数据结构和数据库操作
│   ├── routes/           # 路由层，定义 API 端点和控制器映射
│   ├── services/         # 服务层，封装核心业务逻辑
│   └── utils/            # 工具函数，存放通用的辅助方法
├── views/                # 视图模板文件（如果使用服务端渲染）
├── .gitignore            # Git 版本控制忽略文件
├── app.js                # 项目入口文件，初始化 Koa 应用
├── package.json          # 项目配置和依赖信息
└── README.md             # 项目说明文档
```

### 🏗️ 架构分层说明

这种结构清晰地划分了不同模块的职责，形成了一个多层架构：

1.  **路由层 (routes/)**
    *   **职责：** 定义应用程序的所有 API 端点（URL 路径和 HTTP 方法）。它的作用类似于一个“前台接待”，接收所有来自客户端的请求，并根据请求的路径和方法，将其分发给对应的控制器进行处理。
    *   **示例：** `routes/user.js` 文件中会定义 `GET /users` 和 `POST /users` 等路由。

2.  **控制器层 (controllers/)**
    *   **职责：** 作为路由和具体业务逻辑之间的桥梁。它负责接收路由分发来的请求，从请求中获取参数（如 `ctx.params`, `ctx.query`, `ctx.request.body`），然后调用服务层的方法来处理业务，最后将处理结果（数据或错误信息）组装成 HTTP 响应返回给客户端。
    *   **示例：** `controllers/userController.js` 中的 `getUsers` 函数会调用 `userService.getAllUsers()` 来获取用户列表。

3.  **服务层 (services/)**
    *   **职责：** 这是整个架构的核心，封装了所有的业务逻辑。它不关心 HTTP 请求的细节，只专注于处理数据。例如，处理用户注册、订单计算、数据校验等复杂的业务流程。服务层会调用模型层来完成数据的持久化操作。
    *   **示例：** `services/userService.js` 中的 `createUser` 函数会处理密码加密、数据验证等逻辑，然后调用 `UserModel.create()` 来保存用户。

4.  **模型层 (models/)**
    *   **职责：** 负责与数据库进行直接交互。它定义了数据模型（Schema），并封装了对数据库的增、删、改、查（CRUD）等基础操作。
    *   **示例：** `models/user.js` 文件会定义用户的字段（如用户名、邮箱），并提供 `find()`, `save()`, `remove()` 等方法。

5.  **中间件层 (middleware/)**
    *   **职责：** 这是一个横切关注点，可以在请求到达控制器之前或响应返回客户端之后执行一些通用任务。
    *   **示例：** 身份验证（检查用户是否登录）、日志记录（记录每个请求的详细信息）、错误处理（统一捕获和处理应用中的异常）等。

通过这种分层架构，代码的职责变得非常清晰，每个部分都专注于自己的任务。这使得项目更易于开发、测试和维护。当需要修改某个功能时，开发者可以快速定位到相应的层级，而不会影响到其他部分。
