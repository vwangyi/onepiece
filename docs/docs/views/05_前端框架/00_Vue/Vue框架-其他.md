 template和render函数的区别：

1）template适合简单的组件封装，render函数适合复杂的组件封装

2）template理解起来相对简单，但灵活性不高，性能低，而render灵活性较高，对使用者要求亦高

3）render函数渲染没有编译过程，相当于把代码直接给程序，所以容易出现错误，对使用者要求高

4）render函数较template优先级别高

不管是template语法糖 或 jsx 语法糖 最终都会转为 render函数 



封装组件用template还是render函数？？



 



Vue中render函数和template的区别

vue中一般 使用template来创建HTML,有时也会使用JavaScript来创建HTML，这时就需要render函数

render函数中有一个参数叫h也就是creatElement,render函数将createElement的返回值放到HTML中

createElement有3个参数：

第一个参数（必写参数，类型是字符串，对象，函数）主要用于提供DOM的html内容

第二个参数（可选，类型是对象）主要用于设置这个DOM的一些样式。属性，传的组件的参数，绑定事件之类的

第三个参数（可选，类型是数组）主要是指改节点下还有其他节点

### 问：ref 和 reactive 区别
ref与reactive 是 Vue 3 新推出的主要 API 之一，它们主要用于响应式数据的创建。

1、在 Vue 3 中 reactive 能做的，ref 也能做，reactive 不能做的，ref 也能做。

2、ref 函数创建的响应式数据，在模板中可以直接被使用，在 JS 中需要通过 .value 的形式才能使用。Reactive操作数据与读取数据：均不需要.value。

3、ref 函数可以接收原始数据类型与引用数据类型。reactive 函数只能接收引用数据类型，比如对象。

4、ref 底层还是使用 reactive 来做，ref 是在 reactive 上在进行了封装，增强了其能力，使它支持了对原始数据类型的处理。

### 问：h 函数和createVnode 的区别
开发者常用的是 h 函数，可以用 createVnode 但不推荐

“h() 和 createVNode() 在功能上都是用于创建虚拟DOM节点的，绝大部分情况下可以互换。但它们的主要区别在于：

抽象层级和意图不同：h() 是一个更友好、公共的API，名称源自 Hyperscript，提供了参数规范化等便利功能。而 createVNode() 是一个更底层、内部的API，名称直接描述了其功能。

参数处理不同：这是核心区别。h() 会智能地处理参数，允许省略 props 参数，对开发者更宽容。而 createVNode() 要求参数严格符合标准。

使用场景不同：我们开发者通常直接使用 h()。而Vue内部的模板编译器生成的代码会使用 createVNode()，因为它能保证参数的规范性，无需 h() 的容错开销。

可以简单理解为 h() 是 createVNode() 的一个用户友好的包装器。在Vue的生态中，h() 是写给开发者看的，而 createVNode() 是写给编译器和机器看的。”

h 函数内部会调用 createVNode 

h 函数的好处：一个简短的名字会更加省力 、允许省略 props 参数 

核心结论：开发者常用且推荐的是 h() 函数，但技术上直接使用 createVNode() 也是可行的。

不过，我们需要更深入地理解为什么“可以用”但“不推荐”，这能体现出一个开发者的成熟度。

为什么开发者应该首选 h()

更好的开发者体验 (DX)： h() 的设计初衷就是让开发者用的。它能智能处理参数，允许你写更简洁、更不易出错的代码。

// 更简洁，不易出错

h('div', 'Hello World');

h('div', [h('span')]);

// 更繁琐，容易出错（必须显式写null）

createVNode('div', null, 'Hello World');

createVNode('div', null, [createVNode('span')]);

稳定的公共API： h() 是 Vue 官方公开并承诺在稳定版本中保持兼容的公共 API。它的行为是稳定的，Vue 团队会尽力避免破坏性更改。

明确的意图： 在代码中看到 h()，任何 Vue 开发者都能立刻明白这是在创建虚拟节点，代码的可读性更高。

社区共识： 几乎所有教程、文档和第三方库的例子都使用 h()，遵循共识可以降低团队的协作成本。

为什么 createVNode() “可以用”但“不推荐”

它是底层内部API： createVNode() 的定位更像是 Vue 内部自己的工具函数。虽然它被导出了，但 Vue 团队对其的变更可能更自由，理论上未来发生破坏性变更的风险比 h() 稍高一些（虽然很小）。

更严格的参数要求： 它不具备 h() 的容错能力，要求开发者自己对参数格式负责，这反而增加了犯错的可能性，违背了使用高级抽象（如Vue）的初衷——提升开发效率。

微乎其微的性能收益： 有人可能会觉得少一层函数调用（h 内部会调用 createVNode）会有性能优势。但在现实中，这种差异是完全可以忽略不计的。为了这点微不足道的性能而牺牲代码的简洁性和健壮性，是典型的“过度优化”和“预支负面收益”。

暴露内部细节： 使用 createVNode() 会让你的代码看起来更像 Vue 编译器的输出，而不是手写的业务逻辑，这在一定程度上破坏了代码的抽象层级。

何时可以考虑直接使用 createVNode()

虽然不推荐，但在极少数非常特殊的场景下，你可能会意识到它的存在：

构建极致的性能敏感型库： 如果你在开发一个非常底层的、对性能要求极其苛刻的库（例如另一个前端框架），你可能会选择直接使用 createVNode() 来避免任何额外的函数调用开销。但这对于99.9%的应用开发来说都是不必要的。

与 Vue 编译器行为保持一致： 如果你在做一个工具，需要生成和 Vue 模板编译器完全一致的代码，你可能会选择输出 createVNode 调用。

对于日常业务开发，这些场景几乎不会遇到。

面试时的精彩回答

如果面试官问到这个问题，你可以这样回答，展现你的深度：

“是的，您说得对。从技术上讲，createVNode() 和 h() 都被导出了，确实都可以使用。但在实际开发中，我们团队会严格遵循使用公共API的最佳实践。

h() 是 Vue 设计给开发者的友好接口，它提供的参数规范化能让我们写出更简洁、更健壮的渲染函数，大大改善了开发体验。

而 createVNode() 更像是Vue内部和编译器使用的底层原语。直接使用它意味着我们要手动处理参数格式，放弃了框架提供的便利性，却换不来任何有实际价值的收益，反而引入了不必要的复杂性和极低概率的维护风险。

所以，我们的原则是：除非有极其特殊且充分的理由，否则一律使用公共API h()。这样能保证代码的可读性、稳定性和可维护性。”

这样的回答表明你不仅了解API的区别，更深刻理解软件工程中抽象、稳定性和团队协作的重要性，会给面试官留下非常好的印象。
