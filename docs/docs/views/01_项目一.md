# 埋点耦合业务
## 

这是前端监控领域的经典问题。埋点代码散落在业务逻辑中，会导致**代码臃肿、难以维护、业务变更时容易遗漏埋点**。

## 常见问题表现

```javascript
// ❌ 耦合严重：埋点和业务混在一起
function handleBuy(product) {
  // 业务逻辑
  if (product.stock <= 0) {
    showError('库存不足')
    return
  }
  
  // 埋点代码穿插其中
  track('buy_click', { productId: product.id, price: product.price })
  
  await addToCart(product)
  
  // 又一个埋点
  track('add_cart_success', { productId: product.id })
  
  showSuccess()
  
  // 第三个埋点
  track('buy_success', { productId: product.id, time: Date.now() })
}
```

## 解决方案

### 方案1：AOP 切面编程（推荐）

```javascript
// 封装装饰器/高阶函数
function withTracking(fn, eventName, getParams) {
  return async (...args) => {
    // 前置埋点
    track(`${eventName}_start`, getParams?.(...args))
    
    try {
      const result = await fn(...args)
      // 成功埋点
      track(`${eventName}_success`, { ...getParams?.(...args), result })
      return result
    } catch (error) {
      // 失败埋点
      track(`${eventName}_error`, { ...getParams?.(...args), error: error.message })
      throw error
    }
  }
}

// 业务代码变得干净
const handleBuy = withTracking(
  async (product) => {
    if (product.stock <= 0) throw new Error('库存不足')
    await addToCart(product)
    showSuccess()
  },
  'buy',
  (product) => ({ productId: product.id, price: product.price })
)
```

### 方案2：事件总线/发布订阅

```javascript
// eventBus.js
import mitt from 'mitt'
export const emitter = mitt()

// tracking.js - 监听业务事件
emitter.on('product:addToCart', (params) => {
  track('add_cart', params)
})

emitter.on('product:buy', (params) => {
  track('buy_click', params)
})

// 业务代码 - 只发送事件
function handleBuy(product) {
  // 只发事件，不管谁监听
  emitter.emit('product:buy', { productId: product.id })
  
  // 纯业务逻辑
  await addToCart(product)
  showSuccess()
}
```

### 方案3：声明式埋点（Vue 指令）

```javascript
// 自定义指令 v-track
app.directive('track', {
  mounted(el, binding) {
    const { event, params } = binding.value
    el.addEventListener('click', () => {
      track(event, params)
    })
  }
})

// 模板中使用 - 业务无感知
<button 
  v-track="{ event: 'buy_click', params: { productId: product.id } }"
  @click="handleBuy"
>
  购买
</button>
```

### 方案4：数据层拦截（适合页面浏览埋点）

```javascript
// 拦截 router 跳转
router.beforeEach((to, from) => {
  track('page_view', {
    page: to.name,
    from: from.name,
    timestamp: Date.now()
  })
})

// 拦截 fetch/axios
axios.interceptors.response.use(
  (response) => {
    track('api_success', {
      url: response.config.url,
      duration: Date.now() - response.config.metadata.startTime
    })
    return response
  },
  (error) => {
    track('api_error', { url: error.config.url, message: error.message })
    return Promise.reject(error)
  }
)
```

### 方案5：数据仓库模式（适合复杂场景）

```javascript
// 集中管理所有埋点配置
const trackingConfig = {
  buy_click: {
    event: 'buy_click',
    getParams: (context) => ({
      productId: context.product.id,
      userId: context.user.id,
      timestamp: Date.now()
    })
  }
}

// 统一调用入口
function trackEvent(eventKey, context) {
  const config = trackingConfig[eventKey]
  if (config) {
    track(config.event, config.getParams(context))
  }
}

// 业务代码
function handleBuy(product) {
  trackEvent('buy_click', { product, user })
  // 业务逻辑...
}
```

## 方案对比

| 方案 | 适用场景 | 业务侵入度 | 维护成本 |
|------|----------|------------|----------|
| AOP 切面 | 异步操作、API调用 | 低 | 中 |
| 事件总线 | 多模块、跨组件通信 | 低 | 中 |
| 声明式指令 | UI交互埋点 | 极低 | 低 |
| 数据层拦截 | 路由、请求埋点 | 无侵入 | 低 |
| 数据仓库 | 大型项目、需统一管理 | 中 | 高 |

## 最佳实践组合

```javascript
// 分层处理不同类型埋点
// 1. 页面浏览 - 路由拦截（无侵入）
router.beforeEach((to) => track('page_view', { page: to.name }))

// 2. 用户交互 - 自定义指令（低侵入）
<button v-track:click="'buy_click'">购买</button>

// 3. 业务操作 - AOP 装饰器（中侵入，但规范）
const addToCart = withTracking(async (product) => {
  // 纯业务
  await cartAPI.add(product)
}, 'add_to_cart')

// 4. 异常 - 全局捕获（无侵入）
window.addEventListener('error', (e) => track('js_error', { message: e.message }))
```

**核心原则：通过声明式、拦截器、装饰器等方式，让埋点代码与业务代码解耦，业务开发者不需要关心埋点逻辑。**
