# 前端开发规范文档


1. demo  ：涉及页面 组件 和 store 怎么写  私有组件 公共组件


2. 规范 10-20条 核心规范 也包含demo里面的

1. 圈复杂度不要过高
2. 如何分模块 比如 拆分组件 拆分store   高内聚低耦合  按业务功能拆分
3. views只允许引用自己的私有组件 和 公共组件 不允许引用其views的私有组件。私有组件放在自己的components下
3. 单向数据流
4. 保持低圈复杂度 圈复杂度值必须小于10, 大于10必须重构。
5. 文件代码超过500行 必须拆分
6. 变量声明 无脑const 修改用let 不用var
8. 遵循Vue框架思想，数据驱动DOM, 不要手动操作DOM。
9. for循环 
10. 采用pinia统一管理ViewObject视图对象。不允许用provide/inject跨组件通信。provide/inject只用于封装跨代的父子组件。
10. store只能有一个，不允许用 reactive() 或 hooks fucntion useStore() {} 模拟store。统一采用pinia 。
11. 组件分类 基础组件 业务组件 布局组件
12. 统一使用 script setup  不允许用 vue2写法或optinosAPI
12. 统一优先使用 template写法 非必要不用jsx写法。
13. 防抖节流统一用 lodash 提供的
14. no else 能不用else就不用

1. 单一数据源
```txt 
应用程序的全局状态作为对象存储在单个 store 中。任何给定的数据片段都应仅存在于一个位置，而不是在许多位置重复。

注意：这并不意味着应用中的 所有 状态都必须放进 store 管理！应该根据需要的位置来决定一段状态是属于 store 还是属于你的 UI 组件。

一般来说
需要渲染的核心业务数据都应该放进store
像一个loading变量 属于组件自己管理  
```

## 解决方案
1. 业务复盘会：在业务不忙的情况下，每一周 或 每两周
2. code review: 大家一起审查 前一周或前两周的 提交代码，形成团队规范文档 1234。
3. 具体指标数据是什么 比如 LSP 多少 

总结： 
1. 定期开会，开会2件事：
    1. 业务复盘，大家都要了解其他模块是怎么做的 大家要了解整体业务。
    2. 代码审查：看其他模块负责人如何写代码 好的就学习 不好的就修改，形成统一规范。
2. 已经用了 provide/inject 
可做轻重构，不要重重构
