1.1什么叫进程
程序运行需要内存空间 存放 变量 函数

每个应用至少占用一个进程 进程之间相互独立 互不影响 比如qq和微信和王者荣耀 即使要通信 也要双方同意

什么是线程
进程就可以运行代码 但是谁去读代码 谁去执行代码呢？ 这就是线程

一个进程里面至少有一个线程去run代码 这个线程叫主线程  
如果程序需要同时执行多块代码 主线程可以创建更多的线程去run代码 主线程和其他线程共享一个进程的内存空间
主线程结束 整个程序就结束了

（初步理解 进程是一个环境 线程是一个执行者 一个进程内有多个线程 类似现实生活中的 家庭和家人）

以王者为例子：
主线程负责游戏的界面展示 用户的点击事件 游戏的逻辑处理 游戏的渲染 游戏的音乐播放等等  
 （发现要做的事太多 创建了一个网络线程 专门处理网络相关 创建一个xx进程 专门处理xx事情 即使主线程可以创建多个线程帮忙 但是渲染主线程即使很忙也做不到创建子线程来帮忙）
游戏进程 负责xx
网络进程 负责xx
这些进程可以同时运行各自的代码 互不影响 但是他们共享一个进程的内存空间

2 浏览器的进程模型
浏览器是一个app是一个应用程序和王者荣耀一样 但是它的复杂程度接近操作系统
浏览器是一个多进程多线程的应用程序
为什么是多进程？
为了避免相互影响 减少连环崩溃的几率
当启动浏览器之后 它会自动启动多个进程
主进程 浏览器进程 网络进程 渲染进程 子进程、子进程、、、、 多个进程之间是独立的内存 互不影响 比如网络进程崩溃了 不会影响渲染进程 断网了也可以正常渲染页面
浏览器进程 界面展示（负责浏览器的标签页 前进后退 浏览器皮肤 url地址栏等等 这个界面不包含页面的展示 页面展示统一给渲染进程完成）
用户交互（用户点击等等 ）
子进程管理（创建网络进程 创建渲染进程 销毁网络进程 销毁渲染进程 一开始只有浏览器进程 浏览器可以创建和销毁子进程）
也就是说 浏览器进程里面会有多个线程去处理不同的任务（界面展示是一个线程 用户交互是一个线程 子进程管理是一个线程）

    渲染进程（是浏览器进程创建出来的一个子进程 负责页面渲染的 很重要 前端代码都是渲染进程上的渲染主线程run的）
      首先渲染进程会启动后 会开启一个渲染主线程 用来渲染页面 渲染主线程负责执行HTML CSS JS代码
      默认情况下 浏览器会为每个标签页开启一个新的渲染进程 以保证不同标签页之间不相互影响
      （多学一招： 由于用户经常开启很多个标签页 目前这些标签页都有一个独立的进程 现在是一个标签页就是一个进程 后续可能更改为 一个站点为一个进程 比如2个标签页都是掘金的不同文章 那就是一个进程  但现目前即使是同一个站点 只要是2个标签页就是2个进程）


    事件循环发生在渲染主线程上
    渲染主线程：需要run代码 有 html css js
      1 解析html字符串
      2 解析CSS
      3 计算样式
      4 布局 几何信息（计算每个元素宽高大小 位置坐标）
      5 处理图层
      6 每秒把页面画60次 60帧/秒 60fps  一秒钟画60次
      7 执行全局JS代码
      8 执行事件处理函数
      9 执行计时器的回调
      10 执行微任务
      11 执行宏任务
      12 渲染页面

      多想一步：为什么渲染主线程不启动多个线程来处理呢？ 因为做不到 暂时不理解为啥做不到

    渲染主线程要处理很多事情 而且还遇到一个难题：如何调度任务 （执行任务的顺序是怎样的）
    渲染主线程想到了一个办法： 任务队列 也叫消息队列也叫事件队列 也就是生活中的排队 去医院人多 那就排队

    任务产生来自于各种地方：比如说 用户点击 定时器到时间了  网络请求回来了  promise的resolve reject调用了 执行当前任务的时候 产生了新的任务 等等
    不管任务是从哪里来的 任务都会被放到消息队列中进行排队 渲染主线程只做一件事就是从消息队列中取出第一个任务执行
    比如定时器到时间了 对应的计时线程就会把回调函数放到消息队列里面排队 等待渲染主线程的调度执行
    比如用户点击了一个按钮 对应的交互线程就会把回调函数放到消息队列里面排队 等待渲染主线程的调度执行
    只要是主线程该做的事 其他线程都会帮忙把任务放在消息队列里面排队 等待渲染主线程的调度执行
    渲染主线程只做一件事就是从消息队列中取出第一个任务执行 （就好像那些高学历的医生一样 都是患者来排队 等待医生的调度执行）

    既然是重复的拿任务 那就是写了一个无限循环 只要有患者 就一直拿任务执行
    浏览器源码就是 c++写的 for(;;) { 拿任务执行 }
    消息队列就是一个数组 其他线程负责往这个数组末尾添加任务 渲染主线程负责从这个数组的头部取出任务执行

    1 在最开始的时候 渲染主线程就会进入一个无限循环
    2 每次循环都会检查消息队列中是否有任务存在 有就取出第一个任务执行 没有就进入休眠状态 等待消息队列中有任务时再唤醒
    3 其他线程可以随时向消息队列这个数组末尾添加任务 渲染主线程会无限循环取出第一个任务执行 有就执行 没有就休眠
    这个过程就是事件循环 事件循环是一个无限循环


    自己的话回答事件循环
    首先事件循环是一个无限循环 发生在浏览器的渲染主线程上
    渲染主线程非常忙但其实也就是一件事 就是从任务队列中取出第一个任务执行
    而其他线程负责往任务队列中添加任务 任务队列其实就是一个数组 其他线程负责往这个数组末尾添加任务 渲染主线程负责从这个数组的头部取出任务执行

#### 若干解释

什么叫异步任务？
代码执行过程中 会遇到一些无法立即执行的任务 这些任务叫异步任务
比如 ：计时完成后需要执行的任务 setTimeout(fn,3000) setInterval(fn, 3000)  
 网络通信完成后 需要执行的任务 XHR fetch  
 用户触发后 需要执行的任务 addEventListener('click', fn) DOM.onclick = fn
Promise.then() 等等

    再比如 执行代码遇到了 setTimeout(fn, 3000) setTimeout这个函数会立即执行 fn不会立即执行 会等待0秒后才执行
    setTimeout会通知计时线程开始计时 3秒后结束任务计时线程把fn放到消息队列里面排队 等待渲染主线程的调度执行、


    浏览器采用异步的方式执行任务 为了保证渲染主线程永不阻塞 （如果采用同步的方式执行任务 渲染主线程就会阻塞 比如非要等网络请求回来在执行下一步）

    当渲染主线程遇到一个setTimeout(fn, 3000) 会立即执行setTimeout 并且通知计时线程开始计时 这个任务就已经结束了 渲染主线程继续执行下一个任务
    当计时线程计时结束后 计时线程会把fn放到消息队列里面排队 等待渲染主线程的调度执行

    面试题: 如何理解js的异步？
    参考答案： 浏览器环境的js是单线程的语言 因为js运行在浏览器的渲染主线程上 而渲染主线程有且仅有一个 （注意浏览器是多线程的应用程序 仅仅是js是单线程的）
             渲染主线程有很多事要做 比如说 解析html字符串 css样式表 渲染页面 执行js等等 （尽量说完）
             如果浏览器采用同步的方式执行任务 当渲染主线程遇到 计时器setTimeout 或网络请求 就会等这个异步任务执行完毕 再做下一件事
             而等待的时间就是渲染主线程阻塞的时间 会造成页面卡死现象

             所以浏览器只能采用异步的方式执行任务 为了保证渲染主线程永不阻塞  具体做法是
             当渲染主线程遇到计时器setTimeout 或网络请求 立即通知 其他线程去计时 去请求 计时结束或请求结束
             其他线程把回调放在消息队列里面末尾排队 等待渲染主线程的调度执行

             浏览器采用异步方式执行任务 使渲染主线程永不阻塞 保证了单线程的流畅运行 充分利用了渲染主线程的工作时间 不让渲染主线程空闲

      js为什么会阻塞渲染？
      DOM.onclick = function() {
        h1.textContent = '你好，世界'
        delay(3000) // 这是一个阻塞的函数
      }
      执行效果为 点击按钮 卡3秒 显示 你好，世界
      为什么会卡3秒呢？ 因为js是单线程语言 运行在浏览器的渲染主线程上
      当h1.textContent = '你好，世界'赋值完毕后 内存中确实是 '你好，世界' 然后交给渲染进程去渲染 这个赋值操作已经完成 而渲染任务也是异步任务
      浏览器的渲染主线程去执行了一个 阻塞函数 3 秒 然后才去事件队列中取出渲染任务执行

      这就解释了 js的执行 会影响 浏览器的绘制 因为js是单线程 执行js就不能执行渲染任务 执行渲染任务就不能执行js  渲染任务是异步任务会到任务队列中排队


      任务有优先级吗？
      队列里面的任务没有优先级 但是队列有优先级 因为队列不只一个

      在xx版本之前 有2个任务队列分别是 微任务队列 和 宏任务队列
      队列是有优先级的 微任务队列优先级最高 其他队列优先级低 优先执行微任务 再执行宏任务
      也就是说 先把 微队列这个数组里面的任务按排队顺序执行完毕 再执行宏队列这个数组里面的任务按排队顺序执行完毕

      w3c最新标准：
        任务应该分类型 同一种类型必须在同一个队列中 不同的任务可以分属不同的队列 （换句话说 一个队列里面可以有多种类型 但一定要全 也就是说这些类型不能出现在其他队列里）
        所有浏览器必须有一个微队列队列 且微任务优先级最高 其他队列之间的优先级由浏览器自行决定

      chrome浏览器遵循w3c标准 包含但不限于 微队列 交互队列 延时队列 还有很多队列   队列优先级为 微队列 > 交互队列 > 延时队列

      我可以证明交互队列比延时队列先执行吗？
      先把setTimeout里面的回调函数放到延时队列里面排队 然后把addEventListener点击事件的回调函数放到交互队列里面排队
      执行结果为 即使setTimeout里面的函数先添加到延时队列 但是交互队列的优先级高 所以交互队列的回调函数先执行
      这就解释了有 交互队列和延时队列的存在 而且交互队列的优先级高于延时队列


      // 把fn添加到微队列里面排队  可以用Promise MutationObserver process.nextTick等等
      Promise.resolve().then(fn)
      当渲染主线程遇到 Promise.resolve().then(fn) 会立即执行.then()这个函数 渲染主线程拿到回调fn把fn放到微队列里面排队 等待渲染主线程的调度执行
      当渲染主线程遇到 addEventListener('click', fn) 会立即执行addEventListener()这个函数 通知交互线程 当用户点击后 交互线程把fn放到交互队列里面排队 等待渲染主线程的调度执行
      当渲染主线程遇到 setTimeout(fn, 3000) 会立即执行setTimeout()这个函数 通知延时线程计时 当计时结束后 延时线程把fn放到延时队列里面排队 等待渲染主线程的调度执行

      任务队列里面的任务可能是渲染主线程添加 也可以是其他线程添加 但是任务队列里面任务的执行 只能是 渲染主线程执行

      比如 渲染主线程遇到一个网络请求 通知网络线程去请求网络数据 网络线程请求回来后 把回调放到消息队列里面排队 等待渲染主线程的调度执行
      但是网络请求发生断网 网络请求失败 也会把回调放到消息队列里面排队 等待渲染主线程的调度执行

## 事件循环 消息循环

```js
W3C 里面叫 event loop 事件循环
Google浏览器里面叫 message loop 消息循环

1 在最开始的时候， 渲染主线程会进入一个无限循环
2 每一次循环会检查消息队列中是否有任务存在。
  如果有，就取出第一个任务执行， 执行完毕后进入下一次循环。
  如果没有，则进入休眠状态，等待消息队列中有任务时再唤醒。
3 其他所有线程(渲染进程的其他线程、其他进程的线程) 可以随时
  向消息队列添加任务。新任务会添加到队列的末尾
  在添加新任务时，如果主线程是休眠状态，那么会唤醒主线程 继续循环拿任务
整个过程就是事件循环[消息循环] 无限循环
```

## 何为异步?

```js
无法立即执行的任务
异步任务:
1. 定时器的回调函数fn 等时间到了
2. 事件处理函数fn 等用户触发
3. 网络请求的回调函数fn 等网络回来
4. .then(fn) 等前面promise的 resolve reject的调用


setTimeout()
setInterval()
XHR Fetch
addEventListener()
Promise.then()
如果让渲染主线程等这些异步任务执行完 再去执行下一个任务 那就不对
如果一个点击事件 用户一辈子不触发 渲染主线程就等一辈子 造成浏览器卡死 这是不合理的

渲染主线程还有重要任务 画页面 1秒中画60次 60fps 渲染主线程不及时
渲染页面 用户就看不见 用户的感觉就是卡死了

渲染主线程 无论如何不能阻塞的 因为有及其重要的任务 比如渲染页面
一旦遇到异步阻塞 渲染页面无法执行 给用户造成卡死现象



使用 异步 的方式 渲染主线程永不阻塞
异步的方式:
当渲染主线程遇到定时器 通知 计时线程 开始计时 结束任务
[计时线程 查看 计时器是否到时 到时间就把回调函数放到消息队列里面]
渲染主线程 继续执行下一个任务

当渲染主线程遇到事件处理函数 通知 给dom绑定事件函数
[浏览器进程 会监听用户是否触发了事件 触发了 浏览器进程把回调放到消息队列里面]


面试题: 如何理解JS的异步？
答: 我们知道JS是单线程的，JS之所以是单线程 是因为
JS运行在 浏览器中的渲染进程的渲染主线程上  而渲染主线程只有一个
 所以js是单线程的

 渲染主线程 除了解析js 之外 还有很多事情 比如 渲染页面 解析html 解析css

 如果使用同步的方式 就可能导致渲染主线程阻塞 从而导致消息队列中的很多其他任务无法得到执行，这样一来会导致 渲染主线程白白的消耗时间 另一方面导致页面无法及时渲染更新
 给用户造成卡死现象

 所以浏览器为了解决这个问题 就使用了异步的方式
 具体做法就是  渲染主线程遇到异步任务时 把异步任务交给其他线程去处理
 自身立即结束任务的执行 继续执行下一个任务 当其他线程完成时，将之前传递的回调包装成任务，加到消息队列的末尾处， 依次排队等待渲染主线程的调度执行

 在这种异步模式下， 浏览器的渲染主线程永不阻塞，从而最大限度的保证了单线程的流畅运行


```

## 任务有优先级吗

```js
任务没有优先级的 在消息队列中 先进先出 先排队的先执行
但是 消息队列有优先级

在之前的说法中 消息队列分为 微任务队列 和 宏任务队列
优先执行 微任务 再 执行宏任务

根据W3C官网最新的标准:
每个任务都有一个任务类型，同一个类型的任务必须在同一个队列中 不同的任务可以分属不同的队列
在一次事件循环中 浏览器可以根据实际情况从不同的队列中取出任务来执行


浏览器必须准备好一个微队列[microtask queue 微队列是vip] 微队列任务优先于其他队列任务先执行
连绘制都要等微队列执行完毕才能执行


随着浏览器的复杂度增加 W3C不再使用宏队列的说法

在目前的chrome谷歌浏览器中 至少包含以下队列
1. 微队列: 用户存放需要 最快执行的任务 优先级最高
2. 交互队列： 用于存放用户交互后比如点击后 产生的回调任务 优先级高
3. 延时队列：用于存放计时器到达后的回调任务 优先级中
浏览器还有很多其他队列 由于和开发关系不大 所以不做介绍


w3c只定标准 不做事

chrome浏览器官网:
https://html.spec.whatwg.org/multipage/webappapis.html#perform-a-microtask-checkpoint

```

```js
参考答案：
[在w3c叫事件循环 在google浏览器中叫消息循环]
事件循环又叫消息循环,是浏览器渲染主线程的工作方式
在Chrome 谷歌浏览器的源码中 它开启一个不会结束的 for循环
每次循环都从 消息队列中 取出第一个任务执行
而其他任务只需要在合适的时候将任务加入到消息队列末尾即可

过去把消息队列简单分为 宏队列和微队列
这种说法已经无法满足复杂的浏览器环境了
取而代之的是一种更加灵活多变的处理方式

根据w3c官方的最新解释 每个任务都有不同的类型 同类型的任务必须在同一个队列中
不同类型的任务属于不同队列
不同任务队列有不同的优先级
在一次事件循环中 浏览器除了微队列拥有最高优先级外 其他队列的优先级由各自浏览器
厂商自行决定
```
