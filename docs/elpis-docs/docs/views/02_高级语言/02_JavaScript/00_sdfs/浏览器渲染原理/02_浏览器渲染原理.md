# 浏览器渲染原理

```js
渲染 render

浏览器渲染：把html字符串 ==解析为==> 像素信息 这个过程称为渲染

第一个像素点是什么颜色 第二个像素点是什么颜色 第三个像素点是什么颜色
...


```

```js
function render(html) {
  let pixels;
  // 计算第一个像素点的颜色 第二个像素点的颜色 第三个像素点的颜色 。。。
  return pixels;
}

const pixels = render(html); // 得到像素信息
```

```js
网络： 拿html （css、js都在html中）
渲染： 渲染干了什么事

网络线程 拿到html 生成一个渲染任务

8个步骤
1. 解析html字符串 生成dom树 生成cssom树
2. 解析css字符串  样式计算
3. 把dom树和cssom树合并成一个渲染树   布局
4. 把渲染树绘制到屏幕上  绘制   分层
5. 生成图层
6. 图层合成
7. 显示在屏幕上
8. 重复上面的步骤 每次渲染都要执行 这个串行的过程。

```

## 渲染的第一步是解析 HTML。 解析html 发生了 什么 ？ 生成 dom树和cssom树

```js
渲染的第一步是解析 HTML。

解析开始 首先生成 一个 document 对象
遇到 body 生成一个 body 对象  document.body
遇到 div 生成一个 div 对象  document.body.appendChild(div)
遇到 p 生成一个 p 对象  document.body.appendChild(p)

渲染主线程 遇到 link标签[外部css] 交给 预解析线程
预解析线程 生成一个网络请求任务 交给 网络线程
网络线程 下载好css 交给 预解析线程 预解析线程 解析css 交给 渲染主线程
渲染主线程 生成cssom树
所以css不会阻塞渲染主线程解析html的 因为css有预解析线程帮忙

遇到 script标签 同样会交给 预解析线程 下载[可能有外部js]
但是不同的是 渲染主线程会停止解析html 等待预解析线程解析完js
渲染主线程拿到解析后的js 调用v8引擎执行js
因为在预解析线程解析js的时候 js可能会修改 前面已经生成的dom树
所以 执行js会阻塞渲染主线程解析html 是因为 js可能会修改dom树
由于 js会阻塞渲染主线程解析html 所以 script标签一般放在</body>之前

从网络线程得到的 html字符串[包含css和js] 最终生成 dom树和cssom树
浏览器的默认样式 内部样式 外部样式 行内样式 都在cssom树中



```

## 渲染的第二步是样式计算。 样式计算 发生了 什么 ？ 生成 渲染树 [带有样式的dom树]

```js
渲染主线程 会遍历得到的 dom树 从根节点开始 依次为每个节点计算出它最终的样式
称为 computedStyle
在这一过程中 很多预设值会变为绝对值 red变为rgb(255,0,0)
相对单位 会变为 绝对单位 例如 50% 变为 100px  em 会变为 px


```

## 布局得到布局树

```js
布局阶段 会遍历渲染树 从根节点开始 依次为每个节点计算出它的几何信息
几何信息就是 位置和尺寸
比如 节点的宽高 相对包含块的位置




大多情况下 dom树和布局树不是一一对应的
比如: 父元素的高度是auto 那么就是根据子元素的高度来计算的 %单位根据父元素的宽度来计算的
比如 :before 布局树比dom树多一个节点
比如 display:none 布局树比dom树少一个节点

几何信息就是 位置和尺寸

head为啥是隐藏的 因为google浏览器底层是display:none
style script link meta等都是display:none

匿名行盒 匿名块盒 等等 都会导致布局树和dom树 不一一对应




```

## 分层阶段

```
> 之前浏览器是没有这个阶段的
> 产生原因 如果用户每次操作都要重新布局和绘制整个页面 那么性能就会很差
> 所以浏览器引入了分层的概念 使得一些不需要重新布局和绘制的节点可以复用之前的布局和绘制结果
> 浏览器开发工具 -> ...三个点 -> 更多工具 -> 图层

主线程会使用一套复杂的策略对整个布局树中 进行分层
分层的好处 在于 将来某一个层改变后 只需要对这个层进行重新布局和绘制
而不需要对整个布局树进行重新布局和绘制  从而提升效率

滚动条 堆叠上下文 transform opacity 等样式或多或少都会影响分层结果
也可以通过 will-change: transform; 来强制分层

```

## 绘制阶段

```js
主线程会为每个层单独产生绘制指令集 用于描述这一层的内容该如何画出来

完成绘制后 渲染主线程的工作到此为止 剩余步骤交给其他线程[合成线程]完成
```

## 分块阶段

```js
合成线程会将各个层分解成一个个图块 将其划分为更多的小区域
它会从线程池中取出多个线程来完成分块工作

线程池: 一个进程里有多个线程 但是线程是有限的 一个线程相当于一个工人
你需要人 我就分配给你线程 你再要 我就说等一下 别人用完了我再给你
```

## 光栅化阶段

```js
合成线程会将块信息交给GPU进程
GPU会开辟多个线程 以极高的速度完成光栅化 并且优先处理靠近视口区域的块
光栅化的结果是一块一块的位图
```

## 画

```js
合成线程 拿到每个层 每个块的位图后 生成一个个 指引quad 信息
指引会标识出 每个位图应该画到屏幕上的哪个位置 以及会考虑到旋转 缩放等变形transform
transform 变形发生在合成线程 与渲染主线程无关 这就是transform效率高的本质原因

合成线程会把 quad 信息交给 GPU进程 由GPU进程产生系统调用 提交给GPU硬件 完成最终屏幕成像


```
## 总结

```js
渲染主线程:
        1 解析html  生成dom树 cssom树
        2 计算样式   让每个dom得到最终的样式 生成带有样式的dom树
        3 布局      计算每个dom节点的几何信息 位置和尺寸 生成布局树
        4 分层      根据复杂的策略对布局树进行分层 生成分层树 好处就是不用重新布局和绘制整个页面

                  下一步是分层
                  主线程会使用一套复杂的策略对整个布局树中进行分层
                  分层的好处在于，将来某一个层改变后，仅会对该层进行后续处理，从而提升效率。
                  滚动条、堆叠上下、transform、opacity 等样式都会或多或少的影响分层结果，也可以通过will-change属性更大程度的影响分层结果。
        5 绘制     为每个层生成绘制指令集 用于描述这一层的内容该如何画出来  ,

合成线程:
      1 分块      将各个层分解成一个个图块 将其划分为更多的小区域
      2 光栅化    将块信息交给GPU进程 由GPU进程产生系统调用 提交给GPU硬件 完成最终屏幕成像
      3 画        将每个层的每个块的位图 生成一个个 指引quad 信息 交给GPU进程 由GPU进程产生系统调用 提交给GPU硬件 完成最终屏幕成像

```

渲染的第一步是解析HTML。
解析过程中遇到CSS解析CSS，遇到JS执JS。为了提解析效率，浏览器在开始解析前，会启动一个预解析的线程，率先下载HTML中的外部CSS文件和外部的JS文件。
如果主线程解析到link位置，此时外部的CSS文件还没有下载解析好，主线程不会等待，继续解析后续的HTML。这是因为下载和解析CSS的工作是在预解析线程中进行的。这就是CSS不会阻塞HTML解析的根本原因。
如果主线程解析到script位置，会停止解析HTML，转而等待JS件下载好，并将全局代码解析执行完成后，才能继续解析HTML。这是因为JS代码的执行过程可能会修改当前的DOM树，所以DOM 树的成必须暂停。这就是JS会阻塞 HTML解析的根本原因。
第一步完成后，会得到DOM树和CSSOM树，浏览器的默认样式、内部样式、外部样式、行内样式均会包含在CSSOM 树中。

所以 css 需要放到 body标签之前 所以 js 需要放到dom之后  `</body>`结束标签之前 （boddy后面和 结束body之前有什么区别）
