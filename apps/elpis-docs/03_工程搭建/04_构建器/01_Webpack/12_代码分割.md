# 代码分割



## 问：代码分割怎么做的

是什么： 
    主要是通过 optimization.splitChunks配置项 + import()函数 来做代码分割。
    我会将 splitChunks.chunks 设为 all，

    通过 splitChunks.cacheGroups 把 node_modules中的依赖 抽离成 vendors 文件，这一类变动很少，除非升级。
    同时 把业务中复用率高的公共组件和工具函数 抽离为 common文件，避免重复加载。
    对于 路由组件 使用 import() 并且使用魔法注释webpackChunkName给分割的文件命名， 实现路由懒加载。确保首屏只加载必要代码。



    // 体积过大和代码分割 好像没关系 
    如果有特殊需求，比如针对第三方库体积过大，我会结合 webpack-bundle-analyzer 分析后，进一步细化拆包策略。”
    比如 如果分析得出 echart体积过大。 把全量引入 改为 按需引入 或 把全量引入改为 CDN引入

    最后，优化不能凭感觉。我会使用 webpack-bundle-analyzer 插件来可视化分析包体积，查看是否有意外打入的大文件。
    
    同时，结合 Chrome DevTools 的 Network 面板，观察资源加载的时序，确保代码分割策略真正生效（例如确认非首屏资源确实没有在初始化时下载）。
    同时结合浏览器的网络查看 资源加载时序，确保代码是按需加载 而不是在index.html中直接引入。


好处 
    1. 避免依赖重复打包。避免重复依赖：防止多个入口重复打包相同的库（如 react、lodash）。
    2. 实现按需加载。减少首屏加载时间，将代码拆分成更小的块（chunk），在路由触发或用户交互时再异步加载。通过将代码拆分成多个小的 Chunk，让浏览器只下载当前页面必须的代码，
    3. 借助浏览器缓存实现 强缓存和协商缓存 利用浏览器缓存，更新业务代码不需要重新下载第三方库。

具体:
splitChunks.chunks 的取值有 
async 表示处理 import()函数 异步导入
initial 表示处理 import from 同步导入
all 表示 同步和异步都要处理

initial：只拆分同步加载的代码（入口文件）。
async：只拆分异步（import()）的代码（默认值）。
all：既处理同步也处理异步，webpack 会自动优化。 同一个模块 可能同时被 同步导入 和 异步导入 


splitChunks.cacheGroups 可以 继承或重写 splitChunks下的任何选项
```js
export default {
    optimization: {
        splitChunks: {
            chunks: 'all',
            cacheGroups: {
                vendors: {

                },
                common: {

                }
            }
        }
    }
}
```

## optimization.splitChunks.chunks
