## 一个 vendor 包体积过大（10.3 MiB）

```js
// pnpm add -D webpack-bundle-analyzer


// 在文件顶部引入插件
const { BundleAnalyzerPlugin } = require('webpack-bundle-analyzer');

module.exports = {
  // ... 其他配置
  plugins: [
    // ... 其他插件
    new BundleAnalyzerPlugin({
      // 生成一个静态的 HTML 报告文件，而不是启动一个服务器
      analyzerMode: 'static',
      // 报告文件的名称
      reportFilename: 'bundle-report.html',
      // 生成报告后是否自动在浏览器中打开
      openAnalyzer: true,
    }),
  ],
};
```


根据分析报告，你可以采取以下优化措施：
替换大型库：例如，将 200KB+ 的 moment.js 替换为 2KB 左右的 dayjs。
按需加载：确保你只引入了需要的组件或函数，而不是整个库。例如，使用 lodash-es 并配合 Tree Shaking，或按需引入 UI 组件。
检查重复依赖：分析工具也能帮你发现是否有同一个库的多个版本被打包了进来。





比如 loadsh 改为 loadsh-es 

比如 CDN外部引入 


1. 全量引入 改为 按需引入 或 CDN外部引入（

 
按需引入1: 在使用库是 把全量引入方式 改为 按需引入方式，比如 echarts 体积过大，只使用了10个图表，就只引入10个图表。
CDN外部引入：
    CDN地址有3种情况：
        1. 库的官方就已经提供了CDN的地址
        2. 库官方没有提供，第三方提供了CDN的地址 开放出来，免费使用。
        3. 都没人提供就只有自己 自建CDN，将库下载下来 上传自己的服务器


代码分割: 其实就是代码分割。代码分割：比如某个库很大，但首页不会使用。可以分割为单独的一个js，使用 import()实现单独分割。




图片过大： 图片压缩
Nginx服务器：开启Gzip 压缩，这能让 JS 文件体积在传输时缩小 70% 左右
