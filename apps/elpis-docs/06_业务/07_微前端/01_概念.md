# 微前端



## 
1. 单体架构： 前后端不分离
2. 前后端分离架构
3. 后端有 微服务架构 一个网关 多个子服务
4. 前端有 微前端架构 一个基座 多个子应用



技术栈无关

微组件（微模块）

微前端目前还达不到 微模块微组件的效果 只能页面级拆分 不能组件级拆分



方案1: iframe
缺点： url UI 全局上下文隔离 慢

页面刷新 
路由跳转

方案2: single-spa 
基座：导入子应用的生命周期钩子并调用
子应用：export 导出生命周期钩子
Single-spa只是实现了路由和应用入口，其他微前端的问题统统没有实现，
Single-spa也是基于页面集成 达不到微组件微模块的程度 但是sindle-spa解决了iframe的全局上下文隔离的问题


方案3: 基于single-spa的qiankun 
qiankun是基于single-spa的二次开发

好处：css样式隔离、js隔离、路由状态、通信方式、预加载

方案4: 基于文本component的micro-app
基于原生的 web components 封装的
都是 html entry （引入index.html）

方案5: webpack5的模块联邦module federation 
优点：
1. 去中心化的 可实现 微组件微模块级别
缺点：
vite不能用
模块联邦单独做微前端，个人认为有待商榷， 但是可以把 模块联邦 qiankun miceo-app 进行组合，把模块联邦作为代码模块的共享的


微前端是企业的业务慢慢扩大变为巨石应用才上微前端，微前端往往和老旧技术连接的

集成往往是 根据业务模块的集成
