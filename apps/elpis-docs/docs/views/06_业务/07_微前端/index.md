## 什么是微前端


微前端就是 把一个巨大的 单体应用 拆分为 主应用和多个子应用。主应用负责加载和管理子应用。
微前端就是 把一个单体巨石应用 拆分为 主应用和多个子应用。主应用负责加载和管理子应用。



巨石应用的问题是：  
- 技术栈陈旧，难以升级
- 代码库庞大，构建缓慢
- 团队协作冲突频繁
- 部署风险高，小改动需要全量发布
代码库庞大 构建缓慢 技术栈固化 

 微前端可以按业务拆分 


好处：子应用之间相互独立。使用不同的框架 开发 测试 部署 都是独立的， 
可以逐渐升级，比如先把某一个子应用升级为vue3 
 技术栈无关 不同微应用可以使用不同技术栈（React、Vue、Angular 等）  
独立开发 团队可以独立开发、测试、部署自己的微应用  
独立部署 每个微应用可以单独部署，不影响其他部分  
增量升级 可以逐步升级技术栈，不要求一次全部重写  
简单集成 通过统一协议集成，降低团队间耦合度
独立部署 技术栈自由 渐进式升级



遇到问题：
主应用+子应用通信
处理路由和状态管理 
 

想象一个巨大的电商网站（单体应用），它包含了首页、商品详情、购物车、订单中心、用户中心等模块。在传统模式下，这所有功能都写在一个巨大的代码仓库里，由一个庞大的团队共同维护。
微前端架构则是将这个巨大的应用拆解：
首页团队负责“首页微应用”（可能用 React 写）。
交易团队负责“购物车/订单微应用”（可能用 Vue 写）。
营销团队负责“活动页微应用”（可能用 Svelte 写）。
这些独立的“微应用”在运行时被组合在一起，对用户来说，它们看起来仍然是一个完整的、无缝的网站。qiankun（乾坤）就是目前最流行的实现这一架构的框架之一，它充当了“基座应用”（Main App），负责加载和管理这些子应用。

## 问：你们的微前端项目采用monorepo还是multirepo管理

- 因为对接300多个业务系统 采用multirepo架构管理 子应用有对应的业务团队维护。

如果面试官问：“你们的项目是 Monorepo 还是 Multi-repo？”
建议回答：
“考虑到我们项目对接了 100+ 个异构业务系统，且这些系统由集团内 不同的业务部门（如财务、人力、成本） 独立维护，我们采用的是 Multi-Repo (多仓库) 模式。
具体架构是：
主应用 (基座) 拥有独立的 Git 仓库，由我们中台团队维护，负责统一登录、菜单路由和微前端调度。
各个业务子应用 (如 PD 付款、报税系统) 保留在各自业务线的 Git 仓库中，保持 独立开发、独立构建、独立部署。
公共依赖管理：为了避免版本冲突，我们制定了严格的《微前端接入规范》，约定了核心依赖（如 Vue、ElementPlus）的版本范围。对于公共组件，我们通过搭建 私有 npm 仓库，将通用组件库发布为 npm 包供各子应用引用，而不是直接共享代码。
这种模式虽然增加了本地联调的复杂度（需要通过配置 hosts 或代理指向本地子应用），但它完美解决了 权限隔离、独立发布 和 存量系统迁移 的问题，非常适合我们这种大型集团的场景。”
这样回答既体现了你对架构的深刻理解，又展示了你解决实际工程问题的能力。
