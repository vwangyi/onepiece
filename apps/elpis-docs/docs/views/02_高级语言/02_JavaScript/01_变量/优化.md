+ 优化
    - 减少DOM结构数量，避免嵌套过深的DOM结构
    - 使用defer script 非关键脚本
    - 使用 DocumentFragment 批量操作 DOM
    - 压缩 HTML，移除注释、空白符（生产环境）
    - 优化
        * 减少 DOM 节点数量：避免嵌套过深的 DOM 结构（如超过 10 层）。。。使用 Gzip/Brotli 压缩。异步/延迟非关键脚本：
        * 用 defer 替代 async 可以保证执行顺序
        * 关键的CSS 使用内联的 style标签引入
        * 次要的css 使用 link标签 异步加载 网络优化 CDN 
        * 避免CSS选择器 嵌套过多 div > ul > li > a（右向左匹配效率低）
        * 使用 BEM 等命名规范降低选择器权重
        * 避免使用 @import（同步加载，增加阻塞链）。
        * 拆分 CSS 文件按路由加载（如 home.css, profile.css）。按需加载
        * 使用 will-change 提示浏览器优化。.animate { will-change: transform; }
        * 避免频繁修改样式：用 class 切换替代直接操作 style。
            + // 不好element.style.width = '100px'; // 更好element.classList.add('active');
        * 警惕强制同步布局（如读取 offsetHeight 后立即修改样式）。
    - 优化
        * 减少回流（Reflow）：批量读写 DOM（使用虚拟 DOM 或框架的批量更新）。动画使用 transform/opacity（跳过布局阶段）。使用 Flexbox/Grid：比传统浮动/定位布局性能更高。
        * 表格布局（table）和 position: absolute 容易触发全树回流。
        * 减少绘制区域：使用 overflow: hidden 或 contain: paint 限制绘制范围。.container { contain: paint; }
        * 简化图层：避免不必要的 z-index 和 transform（减少图层数量）。
            + 启用 GPU 加速：
            + .accelerate { transform: translateZ(0); }
        * box-shadow 和 border-radius 是昂贵的绘制操作。
        * 优先使用合成属性：动画使用 transform 和 opacity（跳过布局和绘制）。
            + .slide-in { transform: translateX(100px); }
        * 优化 will-change：仅对实际变化的元素启用：
            + .target { will-change: transform; }
            + 过度使用 will-change 会导致内存占用增加。

            + woker
            + 是什么？浏览器用于 并行处理任务 的后台线程集合（如网络请求、图片解码）。在 Performance 面板中可能显示为 Raster、Compositor 等线程。
            + 作用：观察多线程利用率：如 GPU 光栅化是否饱和。诊断资源加载瓶颈：网络线程是否被占满。
            + 优化手段：避免阻塞线程池：
        * 主线程
            + 主线程是否有长任务
                - 优化 分为小任务
        * GPU
            + 5. GPU是什么？负责 光栅化（Rasterization）和合成（Composition） 的硬件加速单元。在 Performance 面板中显示为 橙色条（如 CSS 3D 变换、`<canvas>` 绘制）。作用：检查 GPU 负载：过度使用可能导致发热/卡顿。验证合成层优化：如 will-change: transform 是否生效。优化手段：优先使用 GPU 加速属性：.animate {  transform: translateZ(0); /* 触发 GPU 加速 */  will-change: transform; /* 提前提示浏览器 */}减少不必要的图层（如过多 z-index）。
