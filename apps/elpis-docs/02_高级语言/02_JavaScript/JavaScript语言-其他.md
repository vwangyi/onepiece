    - 解析URL
        * 如果URL不合法就把输入内容给搜索引擎搜索
    - DNS域名解析
        * 首先判断浏览器或本机hosts文件有没有ip地址 没有就向本地DNS服务器发请求
            + 属于递归请求
        * 本地DNS服务器 会看自身有没有ip地址 没有就向顶级域名服务器和权威域名服务器发请求获取 如果没有提示无法访问
            + 这是属于迭代请求
        * 把域名解析为ip地址
            + 根域名服务器
                - root 简称.     管理顶级域名服务器
            + 顶级域名服务器
                - com top net 
                    * 管理各自的权威域名服务器
            + 权威域名服务器
                - baidu qq bilibili
                    * 管理各自的主机名
            + 主机名
                - www表主站  mail表邮箱 blog表博客 是自定义的
    - TCP三次握手建立连接
        * 客户端向服务器发送 SYN包 （同步包）
            + 表示要同步数据
        * 服务器向客户端发送SYN+ACK包（同步包+确认包）
            + 表示可以同步数据了
        * 客户端再向服务器发送ACK包 （确认包）  此时双方就建立连接了
        * 如果用的是 https 还有多一次TLS握手 用来交换密钥
    - 浏览器发送请求
        * 请求行数据 请求头数据 空行 请求体数据
        * 请求行
            + 协议版本 请求地址 请求方法
        * 请求头

### 问：说一下 重排 重绘 ？
+ 什么是重排：当修改元素的几何属性（比如 尺寸 位置）就会导致页面布局重新排列
+ 什么是重绘：当修改元素的外观样式时（比如颜色、背景等）就会导致重新绘制该元素。
+ 在浏览器渲染过程中 先重排后重绘 所有 重排一定会导致重绘，重绘的性能比重排好
+ 优化：我们应该尽量避免重排重绘 
    - 减少 DOM 操作 把多次操作 DOM 变为 一次
    - 用 transform 代替 left top ，因为 transform 发生在合成线程上 和主线程无关 不会引起重排重绘

### 问：SPA 和 MPA 的区别？
> 单页面应用（SPA）和多页面应用（MPA）在架构、用户体验、性能、SEO、开发难度和适用场景等方面存在显著区别。
>
> 1. 架构差异
>
> 单页面应用（SPA）：只有一个主页面，所有内容通过动态加载和局部刷新实现，依赖前端路由控制页面切换。
>
> 多页面应用（MPA）：由多个独立页面组成，每个页面对应一个HTML文件，通过服务器处理页面跳转。
>
> 2. 用户体验
>
> SPA：提供流畅的交互体验，无需整页刷新，但初次加载时间较长。
>
> MPA：每次跳转都会刷新页面，可能导致短暂等待，但页面加载速度通常较快。
>
> 3. 性能表现
>
> SPA：公共资源只需加载一次，后续交互快速响应，但初始加载资源较多。
>
> MPA：每个页面独立加载必要资源，初始加载速度快，但重复资源可能导致冗余。
>
> 4. SEO支持
>
> SPA：由于内容动态加载，搜索引擎抓取困难，不利于SEO。
>
> MPA：每个页面有独立的URL和固定内容，便于搜索引擎索引，SEO友好。
>
> 5. 开发难度
>
> SPA：需要掌握前端框架和路由机制，页面复杂度较高。
>
> MPA：技术门槛较低，适合初学者，开发和维护相对简单。
>
> 6. 适用场景
>
> SPA：适合交互性强、数据频繁更新的应用，如社交媒体、在线协作工具。
>
> MPA：适合以内容展示为主、需要良好SEO的应用，如新闻网站、企业官网。
>
> 根据项目需求和目标选择合适的架构是关键。如果注重用户交互和流畅体验，SPA是更好的选择；如果需要良好的SEO和简单维护，MPA更为适合。
>

## 为什么移动端设计稿宽度是750px 375px 或375px的倍数?
```javascript

CSS像素(px)、物理像素(pt)

因为iphone6手机屏幕 的物理像素(pt) 宽度为 750px 也就是有750个物理像素点 在制作手机出厂时就确定了


那为什么是375？


480x320像素分辨率

iPhone 6是 `750*1334`个物理像素点 也就是说 在宽度上有 750 个物理像素点
为什么宽度上是750个物理像素点 是苹果制作手机时规定的

普通屏和retina屏(视网膜屏)
在同样大小的屏幕上
假如，普通屏是375个物理像素点，而Retina屏是普通屏的2倍750个物理像素点
Retina屏就比普通屏更清楚 因为Retina屏用2个物理像素点去显示1个逻辑像素点(就是css里面的px)
普通屏 1个物理像素点 去显示 1个逻辑像素点
Retina屏 2个物理像素点 去显示 1个逻辑像素点


同一套代码中 css像素（逻辑像素）都是设置的10px {font-size: 10px}
同一套代码 运行在 普通屏上(假如是375px）那就是1：1的关系  10个物理像素显示10个css像素
同一套代码 运行在 Retina屏上（750px） 那就是2：1的关系  20个物理像素点 去显示10个css像素

疑问:  为什么相同大小的屏幕尺寸 Retina屏的物理像素点可以是普通屏的2倍？
因为 这是苹果发明的技术 同样尺寸大小的屏幕 把更多的物理像素点压缩进去 这个技术是苹果首创的
把屏幕尺寸增大 物理像素点自然就多了 但是手机屏幕尺寸过大 手机握在手上不方便
所以苹果发明了 Retina屏 不增大屏幕尺寸大小的前提下 压缩进去 更多的物理像素点 以达到用更多的物理像素点 去显示css像素 自然就高清了  所以手机的物理像素点就更密集了 因为压缩了更多的物理像素嘛


为什么是750px  因为750px是设计稿的标准宽度  为了方便设计师设计  为了方便前端开发
```
